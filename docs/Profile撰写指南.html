游戏术语抽取 · 项目 Profile 撰写指南

面向项目经理 · 为全新项目准备资料、撰写配置、提高提取准确率

不用写代码填一张表即可准确率可迭代v1 · 2026-06

每个新游戏项目，工具靠一份 Profile（项目配置） 决定"抓什么、不抓什么、怎么分类、怎么译"。本指南告诉你：开新项目时该准备哪些资料、怎么操作、如何把准确率调上去。你不需要懂 YAML——只需填一张表，剩下交给工程/AI 转换。

★速览：一图看懂

原文.xlsx + 术语表.xlsx │ ├─ 自动识别列（原文列 / 中英列） ├─ 去重 → 切块（约1200字，行间重叠防断词） │ ├─ 每批文本，三路"提示"喂给模型： │ ① 术语表命中词 ② AI 扫人名/地名 ③ 百家姓规则兜底人名 │ ├─ LLM 提取 ×3 轮投票（温度 0 / 0.3 / 0.7）→ 3 票全中才保留 │ ├─ 去重 → 去称谓重复（冯大哥→冯）→ 丢通用类 │ └─ 术语表精确命中→直接套译 ／ 没命中→向量找近义参考→AI 翻译 │ ▼ results.xlsx（中英双语术语表）
★ 最关键的一件事

术语表（已有中→英对照）越全，准确率越高。它同时承担三件事：召回兜底、精确套译、近义参考。实测中召回率约 91% 靠术语表撑起来——它比任何 prompt 都重要。能凑多全就凑多全。

PM 实际要交付的

① 游戏文本文件（源文件）
② 尽量完整的术语表
③ 填好的《项目配置填写表.xlsx》
就这三样。工程把 ③ 转成 profiles/<游戏>.yaml，然后跑。

1Profile 是什么

Profile 是一个游戏对应的配置文件（profiles/*.yaml），它把你对这个项目的判断翻译成模型能执行的规则。它控制四件事：

它管什么对应你要回答的问题
抓什么这个游戏里哪些词算"术语"？人名、技能、地名、道具……
不抓什么哪些是噪音？日常物品、泛称、成语、时间词……
怎么分类抓到的术语归到哪些类别？（NPC名 / 武学招式 / 地名建筑……）
怎么译各类术语的翻译方向（人名音译、招式意译……）
✔ 你不用碰 YAML

你只填一张 Excel 表（第 4 节逐题讲）。工程或 AI 把表转成 YAML。本指南里出现的 YAML 只是给你"看一眼成品长啥样"，不要求你手写。

2准备哪些资料

开新项目，按下面三类备齐。第二类（术语表）是准确率的命门，优先级最高。

① 源文件 —— 要被提取的游戏文本

  • 格式：.xlsx，至少有一列是中文原文（要从里面抠术语的那一列）。
  • 可以多个文件分批跑（剧情 / UI / 技能 拆开更好管理）。
  • 列名带"原文 / 文本 / 内容 / 中文"之类，系统能自动认列；列名乱也行，跑前在界面手动确认即可。
  • 变量占位符（${name}、{0}、<color=...>、$N）会被自动剥离，不用你手动清。
  • 双语模式（选用）：如果源文件已经有官方英文列，可同时给中英两列——命中的英文会被原文照抄，省一道翻译且更准。

② 术语表 ★ 命门 —— 已有的中→英对照

★ 为什么它最重要

工具的本质是"术语表负责召回，AI 负责降噪"。术语表里有的词，会被保护不被误删、直接套用已有译文（实测约一半术语靠它直接译）、并作为新词翻译的近义参考。表越全，召回越高、译文越统一。表太薄，再好的 prompt 也救不回召回。

格式：.xlsx，两列——中文术语 | 英文翻译（列名带"中文/术语""英文/译文"可自动识别）。

去哪儿凑（能并就并）：

  • 官方术语库 / 风格指南附带的词表
  • 同 IP、同世界观历史项目的 TM / 术语沉淀
  • 策划案里的：人物表、技能表、道具表、地图/地名表、组织门派表
  • 已定稿批次的术语（边做边回灌，下一批更准）
⚠ 没有现成术语表怎么办

也能跑，但召回会偏低、译文不统一。建议至少手凑一份核心人物 + 核心系统词的小表先垫着，第一批跑完把确认无误的术语回灌进去，边跑边长。

③ 填表素材 —— 用来写 Profile 的背景资料

这些不直接喂工具，而是帮你把《配置填写表》填准。手边备好：

填表需要从哪份资料找
游戏风格 / 世界观一句话立项书、世界观设定文档
要抓 / 不要抓哪些类型客户 brief、翻译需求说明、试译反馈
术语分类清单游戏系统结构（战斗 / 养成 / 社交 / 商城…）
容易搞混的边界情况试译踩过的坑、同类项目经验
正 / 反样例（贴原文标注）真实游戏文本若干段
翻译方向、目标语言客户要求、本地化规范

3操作流程

收集资料

源文件 + 术语表 + 填表素材（见第 2 节）。术语表尽量凑全。

填《项目配置填写表.xlsx》

位于 docs/项目配置填写表.xlsx。只填黄色区，12 题，自然语言作答。逐题讲见第 4 节。

转成 Profile

把填好的表发给工程 / AI，生成 profiles/<游戏>.yaml。你不用手写 YAML。

小样本试跑

界面上传"源文件 + 术语表"，选你的 profile，先只跑 1 个文件 / 一小段看效果，别一上来全量。

审第一批输出

对照结果标两类问题：漏报（该提没提）、误报（不该提提了）。再看分类对不对。

回填迭代

按第 6 节的对照表改 profile / 补术语表，重跑小样本，直到满意。

全量跑 + 交付

满意后跑全量。中断了重传相同文件 + 选相同 profile，会自动断点续跑。输出 results.xlsx 双语术语表。

ℹ 关于"试跑要不要花钱 / 多久"

提取走 3 轮投票，量大时耗时与成本不低，所以务必先小样本验证 profile，再放全量。一次把 profile 调对，比全量跑三遍省得多。

4填写表逐项详解

表分六部分共 12 题。下表把每一题 → 它进入 Profile 的哪个字段 → 怎么写得准串起来。★ 标的是对准确率影响最大的题，请重点投入。

#题目对应字段写法要点
1游戏名称game_type完整中文名。
2游戏风格 / 世界观game_type一句话定调，如"古风武侠开放世界"。模型据此定身份。
3本次任务说明task_description一句话，记录用途。
4★必须提取的类型core_principle + extract_examples.include越详尽越好，逐类列。这是"抓什么"的总纲，直接进系统提示。
5★不要提取的内容extract_examples.exclude列出噪音类型（日常物品 / 泛称 / 成语 / 时间词）。降噪主力。
6★特别注意事项extraction_notes专写边界 / 易混规则，如"老X小X阿X一律人名""排行命名全提取"。
7术语分类列表term_categories每行一类。够用即可，别过细（太细模型分类不稳）。
8目标语言translation_rules英 / 日 / 韩…默认中→英。
9翻译策略说明translation_rules各类译法方向。无特别要求写"统一意译"。
10★应该提取的例子fewshot_examples（正例）贴原文 + 标出哪些是术语、属哪类。覆盖各种类型。
11★不该提取的例子fewshot_examples（负例）贴整句无术语或易误判的原文。负例对降噪极关键，必写。
12其他补充—任何额外说明。

第 4、5 题怎么写得好

✔ 好
「所有人名/NPC名，不论主次、出场次数，出现即提取：2–4字姓名、老X/小X/阿X、X嫂/X爷、戈老大式排行命名、单字动物角色名(青/燕/隼)」
✘ 差
「提取重要的人名」（"重要"无法判定，模型会漏掉路人 NPC，召回崩）
✔ 好
「不提取：无专有名的通用场所(书房/医馆/夜市)、日常材料(柴火/石灰/绳子)、泛称(大侠/公子/弟子)、成语(一飞冲天)、时间词(子时/春分)」
✘ 差
「不要提取没用的词」（无判据，降噪无效）

第 10、11 题——样例是性价比最高的投入

样例（few-shot）是对准确率影响最大的一项：正例提召回，负例提精度。格式随意，标清楚术语+分类即可：

✔ 正例（该提）

「万大海常年往来于沦波坊和神仙渡」
→ 万大海（NPC名）、沦波坊（地名建筑）、神仙渡（地名建筑）

✘ 负例（不该提，整句无术语）

「用木炭生火，草鞋踩在石板上嚓嚓作响」→ 无术语
「长老说弟子们不得擅自出门」→ 无术语（长老/弟子是泛称）

⚠ 负例一定要配几条"整句空结果"

只给正例、不给"这句什么都不提"的负例，模型会倾向多抓 → 噪音飙升。每个项目至少配 3–5 条负例，专挑容易误判的句子。

5为什么这样能提准（原理）

理解每个字段落到哪里、起什么作用，你才知道效果不好时该改哪。

Profile 字段来自第几题进入哪里对准确率的作用
core_principle4系统提示正文高 · 定"抓什么"的总基调
extract_examples.include4系统提示 ✔ 提取清单高 · 提召回
extract_examples.exclude5系统提示 ✘ 不提取清单高 · 降噪
extraction_notes6用户提示·编号规则高 · 处理边界 / 易混
fewshot_examples10 + 11用户提示·示例最高 · 正例提召回，负例提精度
term_categories7用户提示·分类选项中 · 决定分类质量
rule_extractors（百家姓）自动内置代码·分词+姓氏→人名提示人名召回兜底（2–3字姓名）
address_suffixes自动 / 进阶代码·去派生变体精度 · 冯大哥/冯大爷 去重为"冯"
filterable_categories自动 / 进阶代码·按类过滤精度 · 整类丢弃（术语表命中的除外）
术语表（外部文件）资料②命中保护 / 套译 / 向量参考最高 · 召回主力 + 译文统一
translation_rules8 + 9翻译方向记录翻译参考（当前主要作人工与后续译法依据，不直接注入翻译提示）

两条决定成败的机制

① 3 票一致才保留

同一段文本用 3 个不同"随机度"各跑一遍，3 次都提到的词才算数。好处是精度高；代价是——指令含糊会让三轮不一致，词被丢掉（变成漏报）。

对策：规则写得斩钉截铁。用"出现即提取，不论主次"这种绝对口吻，别用"重要的""酌情"。越确定，三轮越一致，召回越稳。

② 术语表 = 召回安全网

AI 真正的角色是"上下文过滤器"——把术语表扫出的大量候选降噪。命中术语表的词被豁免过滤，不会被误删。

对策：发现某些该提的词总被漏 / 被误删，最有效的修法不是改 prompt，而是把这些词补进术语表。

6调优迭代环：看到问题改哪里

第一批跑完，对照下表精确定位。每次只改一两处，重跑小样本验证，别一把全改。

症状首选修法备选
漏报：该提的术语没提 把漏掉的词补进术语表（最有效） 第 10 题加正例；第 4 题 include 写明这类；核对第 6 题规则是否够"绝对"（让三轮一致）
误报：不该提的被提了 第 11 题加负例（含该词的整句→空结果） 第 5 题 exclude 补该类；该类整体没用→设为 filterable_categories
分类错 / 乱 第 10 题样例里把该词标对 category 精简第 7 题分类列表，去掉重叠的类
人名漏（4字以上 / 单字名） 第 6 题写明规则 + 第 10 题给样例 2–3 字姓名已由百家姓自动兜底，无需操心
NPC 称呼重复（冯大哥、冯大爷、冯） 已自动归并为词干"冯" 若该保留全称，把全称加进术语表即可豁免
译文不统一 / 不达意 补术语表（精确命中直接套已有译文） 第 9 题写清该类译法方向
✔ 边跑边长术语表

每跑完一批，把人工确认无误的术语回灌进术语表。下一批的召回、精度、译文统一度都会随之提升——这是收益最高的习惯。

7常见坑 & FAQ

最常见的五个坑

  1. 术语表太薄 → 召回直接崩（它是召回主力）。宁可多塞，错了也比漏了好补。
  2. 只写正例不写负例 → 噪音飙升。负例（整句空结果）必配。
  3. 规则用"重要的""酌情" → 三轮投票不一致，词被丢。改成"出现即提取"这种绝对口吻。
  4. 分类列太多太细 → 模型分类摇摆。够用就好。
  5. 一上来全量跑 → profile 没调对，全量等于烧钱。先小样本。

FAQ

Q：必须有术语表才能跑吗？

不必须，但强烈建议。无表召回会明显偏低、译文不统一。至少手凑核心人物+核心系统词的小表垫着，再边跑边补。

Q：源文件里的 {0}、${name}、<color> 要先清吗？

不用，自动剥离。

Q：列名不规范，系统认错列怎么办？

跑前在界面会显示识别到的列，手动确认 / 改正即可。

Q：跑到一半断了？

重传相同文件、选相同 profile，自动从断点续跑，不重复花费。

Q：百家姓、称谓后缀这些要我填吗？

不用。百家姓人名兜底已内置；称谓后缀是进阶项，默认沿用，特殊需求再找工程调。

附一个完整 Profile 长什么样

仅供参考"成品形态"——你只填表，这步由工程/AI 完成。下面是燕云项目的真实节选（已加注释）：

# 题2 → 游戏定调
game_type: "古风武侠开放世界（燕云十六州）"

# 题4 → 抓什么的总纲（进系统提示）
core_principle: |
  文本中出现的中文人名——无论主线、路人、单次提及——一律提取。
  人物重要性、出场次数、是否主线，都不影响。出现即术语。

# 自动内置：百家姓 2–3 字姓名兜底，无需你填
rule_extractors:
  surname_names: { surnames: "赵钱孙李…", min_len: 2, max_len: 3 }

# 题6 → 边界 / 易混规则
extraction_notes:
  - "老X、小X、阿X、X嫂、X爷、X娘…一个不漏"
  - "通用称谓不提取：先生、大侠、公子、弟子（非专名一部分时）"

# 题4/5 → 提取范围清单（进系统提示）
extract_examples:
  include: ["武学招式", "NPC全名", "门派势力", "排行命名(戈老大)" …]
  exclude: ["日常物品(木炭/草鞋)", "泛称(大侠/公子)", "成语", "时间词"]

# 题10/11 → 正例 + 负例（最影响准确率）
fewshot_examples:
  - { input: "青长老说墨门弟子需恪守门规",
      output: [青长老→NPC名, 墨门→门派势力, 门规→门派系统] }
  - { input: "用木炭生火，草鞋踩在石板上", output: [] }   # 负例：空

# 题7 → 分类清单
term_categories: [武学招式, NPC名, 门派势力, 地名建筑, 道具物品 …]

# 自动 / 进阶：整类丢弃
filterable_categories: [通用物品, 通用NPC]

# 题8/9 → 翻译方向（记录用）
translation_rules: ["角色名→音译", "招式→意译", "地点→意译为主"]
✔ 一句话收尾

资料备齐（尤其术语表）→ 填一张表 → 小样本试跑 → 按症状回填迭代。把这条环跑顺，准确率就稳了。
