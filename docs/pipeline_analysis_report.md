# 游戏术语抽取 Pipeline 分析报告

> 日期：2026-05-19
> 项目：Game-Terms-Extraction-Langlobal（燕云十六州）
> 模型：claude-opus-4-7（主提取）、gemini-3.1-flash-lite（classifier/NER）

---

## 一、分析逻辑链

### 1.1 起点：Precision 过低

初始 pipeline 输出 543 条，Precision 仅 **35.5%**（193 TP / 350 Noise）。噪音集中在：
- 通用日常物品（木炭、草鞋、火把）
- 通用称谓/泛称NPC（弟子、长老、游侠）
- 成语/固定短语（一飞冲天、大鹏展翅）
- 格式异常（游侠$N）

### 1.2 第一轮改进：suffix filter + classifier

| 环节 | 作用 | 效果 |
|------|------|------|
| suffix 称谓派生变体过滤 | 去重 NPC 称呼变体 | 砍 15 条，几乎无 TP 损失 |
| flash-lite 二分类后置过滤 | 术语/非术语判别 | 砍 253 条，误杀 15 TP |

### 1.3 评价盲区的发现

评测过程中发现**三个版本的人工术语表**口径不一致：

| 标准集 | 大小 | 说明 |
|--------|------|------|
| A - 原始人工表 | 457 条 | 包含未在原文出现的词 |
| B - 修订版（cleaned） | 201 条 | 仅保留原文中出现的术语 |
| C - 修订+术语表补充 | 311 条 | B + 全量表命中的 110 条 |

同一份 pipeline 输出在这三套标准上的结果完全不同：

```
         |  基准   | Precision | Recall |  F1  | Noise
A-原始   | 457条   |   60.3%   | 38.3%  | 46.9%| 115
B-修订   | 201条   |   60.3%   | 87.1%  | 71.3%| 115
C-补充   | 311条   |   98.3%   | 91.6%  | 94.8%|   5
```

核心教训：**评价基准的完整度直接决定你对模型效果的判断。**

### 1.4 Classifier 效果的重新评估

对 classifier 删掉的 253 条做交叉分析：

```
Classifier 砍了 253 条:
  ├── 在人工表(cleaned)中:  15  ← 真误杀
  ├── 在全量术语表中:       0   ← glossary safeguard 保护
  ├── 两者皆无:           238  ← 无法验证
```

对 238 条"无法验证"项的人工分类：

```
很可能游戏术语: 134 (56%)  ← 三重神迹、人偶操术、兼爱之道...
模棱两可:       88 (37%)  ← 可能术语可能不是
明显噪音:        16 (7%)  ← 木炭、草鞋、火把
```

→ 结论：**无法判断 classifier 效果好还是不好**，因为缺乏完整的基准真相。已知的是它误杀了 15 条人工标注的 TP。

### 1.5 核心架构的重新理解

比较 pipeline 输出和纯全量表扫原文的结果：

| 方式 | 输出 | Precision | Recall | Noise |
|------|------|-----------|--------|-------|
| 纯全量表扫原文 | 816 条 | 22.4% | 91.0% | 633 条 |
| 当前 pipeline | 283 条 | 61.8% | 87.1% | 108 条 |

→ **LLM 的本质角色不是术语提取器，而是上下文过滤器。** 全量表负责召回（91%），LLM 负责精排（降噪 633→108）。

### 1.6 TP 保护的真实机制

最终输出中的 175 条 TP 分布：

```
TP ∩ 全量表:  174  ← glossary safeguard 保护
TP - 全量表:    1  ← classifier 独立判断保留（老徐）
Classifier 误杀: 15  ← 全量表未覆盖的 NPC 名
```

→ **真正保护 TP 的是 glossary safeguard，不是 classifier。** Classifier 独立判断的 16 条 unprotected TP 中，错判了 15 条。

## 二、Agent 结论

### 2.1 Classifier → 建议移除

- Glossary safeguard 已经承担了全部保护职责
- Classifier 独立决策几乎不贡献价值（仅保 1 条，误杀 15 条）
- 移除后简化维护，消除误杀风险
- Recall 预期恢复 7.5pp（87% → 94%+），Noise 预计增加 ~16 条可接受噪音

### 2.2 Priority 分级 → 当前价值有限

- 投票逻辑中 high 有 2/3 优待
- 但 high 分类的噪音占比高达 25%（47/185），分级不准
- 建议暂时统一为 3/3 硬投票，简化后再考虑恢复

### 2.3 Pipeline 架构定位调整

新理解下的架构图：

```
全量术语表 (28517 条)
        │
        ▼
文本全文扫描 → 816 条候选       ← 低成本，高召回
        │
        ▼
LLM 上下文过滤 → 283 条输出      ← 核心价值：降噪
        │
        ▼
suffix 变体过滤                 ← 辅助
```

## 三、推荐优化方向

### 3.1 方向 A：降本优化（推荐优先）

用弱模型替代 Opus 做过滤，因为现在是分类任务而非生成任务：

| 方案 | 预估成本 | 预估值 Precision | 风险 |
|------|---------|-----------------|------|
| 当前：Opus 提取 | 高 | 61.8% | - |
| Gemini Flash 做过滤 | 极低 | ~55-60%（待测） | 可能降准 |
| Claude Haiku 做过滤 | 低 | ~55-60%（待测） | 可能降准 |
| Rule-based 过滤 | 0 | ~50%（待测） | 规则维护成本 |

**建议实验路径：**
1. 先移除 classifier（零成本改进）
2. 将提取 prompt 改为判别式 prompt，复用已有 816 候选
3. 用 Haiku/Flash 替代 Opus 跑判别，对比 Precision/Recall
4. 按性价比决定最终方案

### 3.2 方向 B：提准优化

如果不考虑成本，从 prompt 层面改进：

- **增加负面示例**：在 profile 的 fewshot 中加入"不提取"示例
- **强化语境判别**：让 Opus 基于上下文判断"这个词在本文中是否是术语"
- **全量表覆盖补充**：将 15 条误杀的 NPC/地名补充进全量表（或单独的白名单文件）

### 3.3 方向 C：混合方案（推荐）

```
全量表扫原文 → 816 候选
        │
        ├── 全量表精确命中 → 直接保留 (~278 条)
        │
        ├── 短词/单字 (≤2字且不在全量表) → Opus/Haiku 判别
        │
        └── 长词 (≥3字且不在全量表) → 规则过滤 + Opus 判别
```

利用全量表的保真性做分层，只在模糊地带调用 LLM，降低成本同时保持精度。
